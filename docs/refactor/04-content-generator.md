
# Phase 4: Build-Time Content Generator

## Goal

Add deterministic build-time discovery for projects and shader/theme modules.

The site must remain static and GitHub Pages compatible. The browser should not scan folders at runtime. Instead, a small Node.js generator script should scan content files and shader modules before development/build, then write generated JavaScript registry files.

After this phase, adding a project or shader should not require editing HTML or CSS.

## Main Problem

After Phase 3, project content is markdown-backed, but the project index is still likely manual.

After Phase 1, shader/theme registration may still be manual.

Manual registries create drift: adding a project or shader requires editing both the source file and the registry. This phase should remove that duplication.

## Hard Constraints

* Keep the site static and GitHub Pages compatible.
* Use vanilla JavaScript, HTML, CSS, Three.js, and a small Node.js build-time script.
* Do not introduce React, Vue, Astro, Svelte, or a backend.
* Do not redesign the website.
* Generated files must be deterministic and safe to commit.
* Do not remove old manual files until generated replacements are verified.
* Do not change project content unless required to satisfy metadata validation.
* Do not change shader visuals unless required by registry integration.

## Scope

Add a build-time generator script that scans:

```txt
content/projects/*.md
```

and the shader/theme folder established in Phase 1.

The script should generate:

```txt
js/projects/ProjectIndex.generated.js
js/background/ThemeRegistry.generated.js
```

Use the actual paths from the repo if Phase 1 or Phase 2 used different names.

Update runtime imports so:

* Project list/detail code reads project metadata from the generated project index.
* Background/theme code reads available themes from the generated theme registry.

Update `package.json` scripts so the generator can be run easily.

## Out of Scope

Do not redesign the site.

Do not migrate project content; that should already be done in Phase 3.

Do not rewrite shader internals.

Do not delete old manual registry files yet unless explicitly safe and approved.

Do not introduce runtime folder scanning.

Do not require a backend.

## Generator Responsibilities

Create a script such as:

```txt
scripts/generate-content-index.mjs
```

The script should:

* Scan project markdown files.
* Parse front matter.
* Validate required project metadata.
* Sort projects deterministically.
* Write `ProjectIndex.generated.js`.
* Scan shader/theme modules.
* Validate shader metadata pattern if possible.
* Sort themes deterministically.
* Write `ThemeRegistry.generated.js`.
* Produce stable output across repeated runs.

## Project Index Generation

The generated project index should include enough metadata for the project list and detail page.

Suggested generated shape:

```js
// This file is generated by scripts/generate-content-index.mjs.
// Do not edit manually.

export const projects = [
  {
    slug: "example-project",
    title: "Example Project",
    summary: "Short summary.",
    date: "2026-01-01",
    type: "Technical Project",
    thumbnail: "assets/projects/example-project/thumb.jpg",
    tags: ["Three.js", "JavaScript"],
    markdownPath: "content/projects/example-project.md",
  },
];
```

Required metadata:

* `slug`
* `title`
* `summary`
* `markdownPath`

Optional but preferred metadata:

* `date`
* `type`
* `thumbnail`
* `tags`

If required metadata is missing, the generator should fail with a clear error.

## Theme Registry Generation

The generated theme registry should include all shader/theme modules.

The exact shape should match the Phase 1 architecture.

Suggested generated shape:

```js
// This file is generated by scripts/generate-content-index.mjs.
// Do not edit manually.

import { themeMeta as oldTvNoiseMeta, createTheme as createOldTvNoiseTheme } from "../shaders/oldTvNoise.js";
import { themeMeta as fluid2dGasMeta, createTheme as createFluid2dGasTheme } from "../shaders/fluid2dGas.js";

export const themes = [
  {
    ...oldTvNoiseMeta,
    createTheme: createOldTvNoiseTheme,
  },
  {
    ...fluid2dGasMeta,
    createTheme: createFluid2dGasTheme,
  },
].sort((a, b) => {
  const orderA = a.order ?? 0;
  const orderB = b.order ?? 0;

  if (orderA !== orderB) {
    return orderA - orderB;
  }

  return a.label.localeCompare(b.label);
});
```

The actual relative import paths must be valid for the generated file location.

## Shader Module Requirements

Each shader module should export:

```js
export const themeMeta = {
  id: "theme-id",
  label: "Theme Label",
  order: 10,
};

export function createTheme(context) {
  return {
    update(time, deltaTime) {},
    resize(viewport) {},
    dispose() {},
  };
}
```

If a shader file does not follow this contract, the generator should either skip it with a warning or fail clearly. Prefer failing clearly if the file is expected to be a theme.

## Package Scripts

Update `package.json` with scripts similar to:

```json
{
  "scripts": {
    "generate": "node scripts/generate-content-index.mjs",
    "dev": "npm run generate && vite --host 127.0.0.1 --port 5173",
    "build": "npm run generate",
    "preview": "vite preview --host 127.0.0.1 --port 4173"
  }
}
```

Adapt the scripts to the repo’s actual existing local server setup. Do not break the user’s existing `.nvim.lua` workflow.

The important requirement is that there is an easy command to regenerate content indexes.

## Determinism Requirements

Running the generator twice without source changes should produce identical generated files.

To ensure this:

* Sort project files by slug or path.
* Sort themes by `order`, then label or id.
* Use consistent line endings.
* Use stable JSON/string formatting.
* Avoid timestamps in generated files.

## Generated File Header

Each generated file should start with a warning:

```js
// This file is generated by scripts/generate-content-index.mjs.
// Do not edit manually.
```

## Implementation Rules

Before editing, inspect the repo and list:

* Current manual project index file.
* Current manual theme registry/config file.
* Current shader/theme module folder.
* Current markdown project folder.
* Existing package scripts.
* Exact generated file paths.
* Exact files to change.

During implementation:

* Add the generator script.
* Generate both registries.
* Update runtime imports to use generated registries.
* Keep manual files temporarily if needed for safe comparison.
* Run the generator at least twice and verify stable output.
* Keep the diff focused on generated discovery.
* Do not redesign the site.
* Do not remove old project pages in this phase.

## Acceptance Criteria

The phase is complete when:

* A generator script exists.
* Project markdown files are scanned.
* Shader/theme modules are scanned.
* `ProjectIndex.generated.js` is generated.
* `ThemeRegistry.generated.js` is generated.
* Generated output is deterministic.
* Project list/detail pages use the generated project index.
* Background/theme system uses the generated theme registry.
* Adding a markdown project and rerunning the generator makes the project available.
* Adding a shader module and rerunning the generator makes the shader available.
* No HTML or CSS edits are required for normal project/shader additions.
* The site remains static and GitHub Pages compatible.

## Test Plan

After implementation, test:

1. Run the generator.
2. Run the generator again.
3. Confirm generated files do not change on the second run.
4. Start the local dev server.
5. Open the home page.
6. Open the projects page.
7. Open every project detail page.
8. Switch through every shader/theme.
9. Add a temporary test markdown project.
10. Rerun the generator.
11. Confirm the test project appears.
12. Remove the test project.
13. Rerun the generator.
14. Add a temporary test shader module if practical.
15. Rerun the generator.
16. Confirm the test shader appears.
17. Remove the test shader.
18. Rerun the generator.
19. Confirm the browser console has no new runtime errors.

After implementation, stop and report:

* Runtime imports changed.
* Package scripts changed.
* How to run the generator.
* How to test determinism.
